## EMR 데스크톱 앱 디자인 원칙 — 업계 베스트 프랙티스 기반

### 1. 공인 프레임워크: AMIA 14 유사빌리티 원칙

AMIA(American Medical Informatics Association)가 JAMIA에 발표한 EMR 설계를 위한 14가지 유사빌리티 원칙이 이 분야의 사실상 표준입니다. Dops에 적용할 핵심 항목을 추리면:

**Consistency (일관성)** — 동일한 색상 체계, 버튼 배치, 라벨링을 전 섹션에 걸쳐 유지해야 직관적 내비게이션이 가능하고, 사용자가 시스템의 다른 부분을 익히는 데 드는 시간이 줄어듭니다.

**Visibility (가시성)** — 사용자는 EHR UI에 표시된 모든 정보를 보고 이해할 수 있어야 하며, 관련 정보를 한 화면에 모으거나 쉽게 접근 가능하게 해야 합니다.

**Minimalism (최소주의)** — 불필요한 단계와 방해 요소를 제거하여 최소 클릭으로 필요한 작업을 완료할 수 있도록 하고, 알림과 리마인더는 생명에 관련된 것만으로 제한해야 합니다.

**Memory Load Minimization (기억 부하 최소화)** — AMA에 따르면 인지적 부담 감소가 EMR 유사빌리티 개선의 최우선 과제 중 하나입니다.

**Error Prevention + Reversibility** — 화면 상태를 명확히 하고, 폼 유효성 검사를 강화하며, 워크플로우 큐를 제공하여 하류 오류를 줄여야 합니다.

### 2. 레이아웃 원칙: 고정 프레임 + 내부 스크롤

인하 님이 이미 정한 "상단 메뉴바 + 하단 디테일 2단 구성"은 업계 베스트 프랙티스와 정확히 일치합니다.

**이중 스크롤 금지** — 이중 스크롤(double scroll)은 컨테이너에 스크롤바가 있으면서 페이지 전체에도 스크롤바가 있는 상태인데, 데스크톱에서도 용납할 수 있는 수준이지만 사용자에게 상당히 답답함을 줄 수 있습니다. EMR처럼 정보 밀도가 높은 앱에서는 반드시 피해야 합니다.

**전체 뷰포트 고정, 각 패널 내부 스크롤** — 이것이 데스크톱 EMR의 표준 패턴입니다. 구체적으로:

- 앱 셸(상단 메뉴바 + 사이드바 등)은 `height: 100vh`, `overflow: hidden`
- 각 콘텐츠 영역(환자 목록, 차트 내용 등)만 개별적으로 `overflow-y: auto`
- Electron에서는 `window.resizeTo` 이벤트에 반응하여 각 패널이 동적으로 높이를 계산

**한 화면에 모든 것을 보여주지 말 것** — 핵심 정보에 집중하고 각 단계에서 관련된 데이터만 보여주되, 다른 EMR/EHR 플랫폼에서 사용하는 익숙한 UI 패턴을 활용해야 합니다.

---

### 3. 역할 기반 대시보드

로그인 시 사용자 유형을 식별하여 해당 역할에 특화된 화면이나 대시보드를 표시해야 합니다. 역할에 따른 서로 다른 UI를 제공하면 더 단순하고 집중된 사용자 경험이 만들어집니다.

Dops의 경우: 원장(진료), 데스크(접수/수납), 위생사(스케일링/예방) 등 역할별 진입 화면이 달라야 합니다.

---

### 4. 통합 방문 뷰 (Unified Visit View)

기존 EHR 시스템의 구조가 진료 중 효율적으로 사용하기엔 너무 복잡하다는 점이 의사들의 지속적인 불만이었습니다. 이를 해결하기 위해 모든 관련 정보를 한 곳에 표시하는 통합 방문 뷰를 도입하여, 의사가 활성 방문 노트, 첨부 문서, 이전 코멘트, 예약, 확정 진단을 컨텍스트 전환 없이 확인할 수 있게 했습니다.

---

### 5. 색상 & 타이포그래피

**색상:** 블루는 신뢰감과 안정감, 그린은 건강과 성장의 연상을 주는데, 의료 분야에서는 전통적으로 의료진 유니폼과 보험 브랜딩에서 파란색을 사용해왔습니다. 의료 색상 체계는 소프트 블루, 그린, 쿨 뉴트럴을 기반으로 하는데, 이들이 문화와 연령대를 넘어서 깨끗하고 안정적이며 전문적으로 읽히기 때문입니다.

구체적 가이드:

- **주색:** Navy/Deep Blue (신뢰·전문성) — 상단바, 주요 CTA
- **보조색:** Teal/Soft Green — 상태 표시, 성공 피드백
- **배경:** Off-white ~ Light Gray (#F8FAFC ~ #F1F5F9) — 눈 피로 감소
- **경고:** Red는 오직 위험/오류에만, Amber는 주의에만 사용
- **대비:** WCAG AA 최소 4.5:1 준수 (정보가 생명과 직결되므로)

**타이포그래피:** 깔끔하고 현대적인 폰트는 전문성과 신뢰감을 전달하며, 가독성과 판독성이 특히 의료 앱에서는 필수입니다. Sans-serif 계열(Pretendard, Inter 등)을 기본으로, 데이터 테이블은 고정폭 또는 tabular figures를 가진 폰트 사용.

### 6. 정보 밀도 관리

EHR/EMR 인터페이스가 어수선하면 사용자가 잘못된 탭을 클릭하거나, 잘못된 필드에 데이터를 입력하거나, 중요한 업데이트를 놓치게 됩니다.

**Negative Space(여백)** — 여유 공간은 콘텐츠를 분리하고, 핵심 데이터를 강조하며, 시각적 과부하를 방지합니다. 낭비되는 공간이 아니라 EHR 디자인에서 가독성을 높이고 사용자 피로를 줄이는 역할입니다.

**데이터 시각화** — 환자의 상태 변화를 그래프와 시각적 표현으로 보여주고, 시계열 데이터를 구현하면 건강 정보의 가독성이 향상되고 변화 추적에 도움이 됩니다.

---

### 7. 클릭 수 최적화

한 건의 환자 노트를 기록하는 데 5개 화면에서 10번 클릭이 필요하다면, 의사들은 시스템을 회피합니다. 워크플로우에서 가치가 낮은 단계를 제거하면 약 20%까지 개선할 수 있다는 연구 결과가 있습니다.

---

### 8. Electron + 웹 기술 적용 시 유의사항

Dops가 Electron 기반이므로 추가적으로:

- **네이티브 윈도우 크롬:** 커스텀 타이틀바 사용 시 드래그 영역, 최소/최대/닫기 버튼 OS 표준을 따를 것
- **키보드 내비게이션:** 데스크탑 앱에서는 Tab 순서, 단축키가 특히 중요 (진료 중 마우스를 쓸 시간이 없음)
- **오프라인 대응:** 네트워크 불안정 시에도 로컬 데이터로 진료 가능하도록 상태 표시
- **성능:** EMR 시스템이 중요한 순간에 지연되거나 멈추거나 충돌하면 워크플로우가 중단되고 환자 안전이 위협받습니다. Electron의 렌더러 프로세스 무거워지지 않도록 가상 스크롤, 지연 로딩 적극 활용

---

### 9. 신뢰성을 만드는 미시적 디자인 요소

색상의 의미를 일관되게 유지하여 색상이 결정을 방해하는 것이 아니라 지원하도록 해야 합니다. 구체적으로:

- **상태 색상 고정:** 초록=완료/정상, 노랑=대기/주의, 빨강=오류/위험 — 전 앱에서 예외 없이
- **로딩/저장 피드백:** 모든 데이터 변경에 즉각적 시각 피드백 (자동저장 인디케이터 등)
- **Undo 지원:** 도움이 되는 빈 상태, 정중한 확인, 필요할 때의 "되돌리기", 부드러운 피드백 모션이 제품을 더 친절하게 만들고 지원 문의를 줄여줍니다.
- **UI Kit 통일:** 스타일 일관성을 유지하기 위해 모든 핵심 인터페이스 컴포넌트를 포함하는 UI Kit를 구성할 것을 권장합니다.
