## 문서의 개요

프로토타입 작업 이후, 작업 내용 요약하여 디자이너에게 전달하기 위한 문서 생성 지침

### 문서의 목적

이 문서는 소프트웨어 기능의 스펙을 전달하는 것이 아니라, **읽는 사람이 현장의 업무 흐름을 체득하게 하는 것**이 목적이다. 문서를 읽은 사람이 프로토타입보다 더 나은 결과물을 만들 수 있어야 한다.

### 독자

도메인 지식이 없는 전문 디자이너와 개발자. 기술적 역량은 높지만 해당 업무 현장을 모른다.

### 서술 원칙

**업무가 먼저, 화면은 나중에.** 화면 설명부터 시작하지 않는다. "왜 이 업무가 존재하는가" → "현장에서 실제로 어떻게 흘러가는가" → "어떤 문제가 있는가" → "그래서 이렇게 설계했다" 순서로 서술한다. 화면은 이 맥락을 이해한 뒤 참고하는 보조 자료이다.

**스펙이 아니라 이야기로 쓴다.** "표시 항목: 서명일시, 서명자, 건수" 같은 나열이 아니라, "대표원장이 월초에 출근해서 제일 먼저 하는 일은..."처럼 사용자의 행동과 맥락 속에서 기능이 등장하게 한다. 읽는 사람이 그 장면을 머릿속에 그릴 수 있어야 한다.

**의사결정의 "왜"를 반드시 포함한다.** 이 대화에서 논의한 선택과 트레이드오프를 기록한다. "대리서명이라는 용어를 쓰지 않는다"만 적지 말고, "법적 근거가 미비한 회색 지대이므로 UI에서 해당 표현을 피한다"까지 적는다. 배경을 모르면 나중에 누군가 "대리서명 버튼 추가해주세요"라고 할 수 있다.

**사용자를 존중하는 톤으로 묘사한다.** 이 앱의 사용자는 고수준의 교육을 받은 전문가다. "사용자가 헷갈릴 수 있으므로"보다 "이 정보를 이미 알고 있으므로 반복 노출은 노이즈가 된다"처럼, 전문가에게 맞는 설계 근거를 서술한다.

**현장 용어를 그대로 쓴다.** 페이닥터, 봉직의, 실장, 개원시작일 등 현장에서 실제로 사용하는 용어를 그대로 쓰고, 필요하면 괄호로 보충 설명한다. 일반화된 용어로 바꾸면 현장감이 사라진다.

### 문서 구조

**1장. 업무 배경** — 이 기능이 왜 존재하는지. 법적 요구사항, 문서의 성격 차이, 병원 내 역할 구조를 설명한다. 읽는 사람이 "아, 이런 세계가 있구나"를 이해하는 단계.

**2장. 실제 업무 흐름** — 현장에서 서명이 언제, 어떻게 발생하는지 시나리오로 서술한다. 빈도, 패턴, 예외 상황(퇴사 후 연락 두절 등)을 포함한다. 읽는 사람이 사용자의 하루를 상상할 수 있는 단계.

**3장. 문제와 설계 의도** — 2장의 업무 흐름에서 어떤 문제가 발생하고, 그것을 어떻게 해결하려 했는지. 화면 단위가 아니라 문제 단위로 서술한다. "잊지 않게 하기", "빠르게 처리하게 하기" 같은 문제 프레임 안에서 여러 화면의 설계 의도가 자연스럽게 설명된다.

**4장. 프로토타입 설명** — 위 맥락을 이해한 상태에서, 프로토타입의 각 화면이 어떤 업무 시나리오에 대응하는지 간결하게 매핑한다. 세부 스펙 나열이 아니라 "이 화면은 이런 상황에서 쓰인다" 수준. 프로토타입은 최종 목표물이 아니라 참고 자료이므로, 더 나은 방향이 있다면 변경할 수 있다는 여지를 남긴다.

**5장. 정책 요약** — 상태 정의, 권한 매트릭스, 용어 규칙, 공통 인터랙션 규칙 등 개발/디자인 시 참조할 사항을 표로 정리한다. 앞장에서 서술한 내용의 요약이지 새로운 정보가 아니어야 한다.

**6장. 기타사항** — 향후 검토 사항, 미구현 기능, 열린 질문 등을 간결하게 기록한다.

### Do

- 이 문서의 목적을 달성하기 위해서라면, 이 문서에 제시되 내용 또는 제시된 정형화된 문서구조 등을 자율적으로 변경하여 작성하라
- 문장 서술은 간결하게. 합니다 체를 사용하지 말고 "~이다" 등으로 간결하게, 또는 요약식으로 명사형 어미 사용.

### Don't

- 화면 필드를 하나하나 나열하는 스펙 문서 형태
- "~해야 합니다", "~을 제공합니다"가 반복되는 요구사항 문서 톤
- 업무 맥락 없이 UI 컴포넌트만 설명하는 서술
- 기술 구현 방법에 대한 언급 (전문 개발진이므로 불필요)
- 디자인 가이드(색상, 폰트, 여백 등) 지정 (전문 디자이너에게 위임)
