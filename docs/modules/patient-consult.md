# Dops 치료계획 화면 — 설계 전달 문서

> 이 문서는 치료계획 화면 프로토타입의 스펙을 나열하는 것이 아니라, **읽는 사람이 치과 현장의 업무 흐름을 체득하고 더 나은 화면을 만들 수 있게 하는 것**이 목적이다. 프로토타입은 최종 목표물이 아니라 참고 자료다. 더 나은 방향이 있으면 바꿔도 된다.
>
> 대상은 도메인 지식이 없는 전문 디자이너·개발자다. 색상·폰트·여백 등 비주얼 디자인은 별도 디자인 원칙 문서에 위임하며, 여기서는 다루지 않는다.

---

## 1장. 업무 배경

### 치료계획이 무엇인가

치료계획은 진단 후 환자에게 제안하는 **전체 시술 묶음**이다. 임플란트 하나가 아니라 "발치 + 뼈이식 + 임플란트 + 크라운"처럼 여러 시술의 집합이며, 대부분 비보험·고가다. 따라서 비용을 환자에게 설명하고 동의를 받는 것이 핵심이다.

치료계획은 단순 견적서가 아니다. 한 엔티티가 네 가지를 겸한다 — 시술 진행 트래커, 수납 트래커, 환자 동의 서명 기록, 그리고 환자 상담의 중심.

### 왜 새로 설계하는가

기존 덴트웹은 "치료계획"과 "치료비용계획"을 **두 개의 분리된 엔티티**로 두었다. 치과의사는 치료계획만 작성하고, 실장(데스크 책임 직원)은 그것을 보며 치료비용계획을 처음부터 다시 입력하는 구조였다. 분업 의도였다.

현실에서는 작동하지 않았다. 차팅을 직원이 전담하는 분위기라 결국 같은 사람이 두 번 입력하는 꼴이 됐고, 환자는 항목별 비교에 관심이 없고 "총 얼마"만 궁금해했다. 결과적으로 치료계획은 사장되고 치료비용계획만 살아남았다.

그래서 Dops는 **두 엔티티를 하나로 통합**한다. 세부옵션·메모·비용을 한 곳에 담는 단일 치료계획이다. 관통하는 철학은 **"느슨한 단일 소스"** — 어설프게 엄격한 UX는 현장에서 회피되므로, 강제와 자동 전환을 최소화하되 일관된 UX를 유지한다.

### 이 화면의 특수성: 환자가 같이 본다

이 화면은 직원만 보는 게 아니라 **환자와 함께 보는** 화면이다(모니터를 돌려 보여주거나, 인쇄해 건넨다). 이 한 가지 사실이 뒤에 나올 여러 설계 결정의 뿌리다.

### 화면 안에서의 위치

전자차트는 좌/우로 나뉜다. 좌측은 차트 내용(종이차트에서 착안한 세로 무한 스크롤)을 보여주고, 우측은 입력 영역으로 여러 탭을 가진다. 이 문서가 다루는 것은 **전자차트 우측의 "치료계획" 탭**이다.

## 2장. 실제 업무 흐름

### 누가 입력하는가

원장이 진료 중 구두로 지시하면 직원이 받아 적는 패턴이 우세하다. 원장이 큰 틀(어떤 시술을 할지)만 잡고, 비용·세부옵션·메모는 직원이 채우는 경우가 많다. **역할별 권한 구분은 두지 않는다.** 누구나 만들고 수정하고 종결할 수 있다. 개원가 치과에서 직원에게 강한 권한을 위임하는 실제 운영에 맞춘 것이다.

### 환자가 궁금한 것은 총액

"알아서 좋은 걸로 해주세요"와 "총 얼마예요?"가 환자의 일반적 반응이다. 항목별로 옵션을 비교하며 상담하는 흐름은 현장과 어긋난다. 그래서 화면은 항목 하나하나의 미세 조정보다 **묶음 전체와 총액**이 잘 읽히는 데 무게를 둔다.

### 상담일과 서명

상담이 이뤄진 날짜를 기록하고, 계획+비용 전체에 대해 환자가 한 번 서명한다(태블릿 사인 또는 종이 스캔). 단, **동의를 아직 못 받았더라도 진료 입력을 막지 않는다.** 전자차트의 본분은 기록이기 때문이다.

### 진행 중 변경과 보류

한 번 만들고 끝나는 경우가 많지만, 진행 중 변경도 잦다(새 병소 발견, 환자 요청 등). 특히 환자가 "이건 빼주세요" 하는 경우가 흔한데, 이때 그 항목은 **보류**가 되고 총액에서도 빠진다. 삭제가 아니라 보류인 이유는, 나중에 다시 할 수도 있고 상담 이력으로 남겨야 하기 때문이다.

## 3장. 문제와 설계 의도

화면 단위가 아니라 문제 단위로 서술한다. 하나의 문제를 풀기 위해 여러 화면 요소가 함께 움직인다.

### 문제 1 — 환자 앞에서 할인을 어떻게 다룰까

할인은 양날의 검이다. 환자는 총액 할인을 기대하고, 운영상으로도 자연스럽다. 그러나 환불 분쟁이 생기면 항목별 금액이 모호해져 법적으로 위험하다.

설계는 두 갈래로 나눠 풀었다.

첫째, **데이터 모델 단계에서 정리한다.** 항목별 할인 필드를 두지 않는다. 가격을 깎고 싶으면 그 시술의 단가를 직접 수정한다(그러면 각 항목의 단가가 곧 그 시술의 가격이라 분쟁 시 명확하다). 별도의 "할인" 개념은 총액 차원에서만 둔다 — 항목 단가는 그대로 두고 총액에서 차감하는 "이번에 특별히 깎아드린 금액"이다.

둘째, **환자 앞 노출을 통제한다.** 소액 할인처럼 굳이 환자에게 "할인했다"를 드러내고 싶지 않은 경우가 있다. 그래서 헤더에 **환자 표시 토글**을 둔다. 토글을 켜면 할인액 줄이 사라지고 계약금액만 남는다. 이때 계약금액에는 이미 할인이 반영돼 있으므로, 환자가 보는 최종 금액은 정확하다. 단, 할인이 실제로 발생한 경우(할인액 > 0)에는 계약금액을 입력하고 빠져나오는 순간 토글이 자동으로 풀려 할인이 다시 드러난다 — 큰 할인은 오히려 "이만큼 깎아드렸다"고 보여주는 게 환자에게 유리하기 때문이다.

### 문제 2 — 입력은 빠르게, 표시는 정확하게

직원은 진료 중 "55만, 135만" 식으로 **만원 단위**로 말하고 적는다. 그런데 환자에게 보이고 계약서로 나가는 금액은 **원 전체(1,350,000원)**라야 신뢰가 간다.

그래서 금액은 입력과 표시의 단위를 분리한다. 평소엔 원 전체로 표시하고, 칸을 누르면 만원 단위로 입력하는 칸이 된다. 0을 다섯 개 칠 필요가 없다. 추가로, 새 항목의 금액을 처음부터 새로 잡을 때를 위해 **우클릭하면 0원으로 비우면서 편집 모드로 들어가는** 단축 동작을 뒀다.

### 문제 3 — 시술 항목을 어떻게 다루게 할까

시술명(크라운, 임플란트 등)은 **바꿀 수 없다.** 바꾸려면 그 줄을 삭제하고 새로 추가한다. 시술명이 바뀌면 연결된 옵션·단가 체계가 통째로 달라지므로, 어설프게 인라인으로 바꾸게 두면 데이터가 어긋난다. 반면 같은 시술 안의 **세부옵션**(지르코니아냐 골드냐)은 자주 바뀌므로, 누르면 그 시술에 해당하는 옵션 목록이 떠 고를 수 있다.

새 항목은 목록 맨 아래의 추가 자리에서 만든다. 종이차트가 시간순으로 아래에 이어 붙는 것과 같은 공간 모델이다. 진료 중 "여기 충치도 있네" 할 때 목록 끝에 한 줄 더하는 동작이 가장 클릭이 적다.

### 문제 4 — 메모를 잊지 않고 빠르게

메모는 상담 중 즉석에서 적는다. 별도 저장 버튼이나 모달을 거치면 흐름이 끊긴다. 그래서 메모는 셀을 누르면 그 자리에서 바로 고칠 수 있다(인라인). 다만 편집에 들어가고 나갈 때 **줄 높이가 변하거나 표가 흔들리면 안 된다.** 정보 밀도가 높은 화면에서 레이아웃이 출렁이면 피로가 크기 때문이다. 그래서 편집 전환은 크기·위치를 그대로 두고 배경색만 바뀌는 방식으로 처리했다.

### 문제 5 — 모든 정보를 시간순으로, 같은 언어로

좌측 차트가 종이차트식 무한 스크롤이듯, 치료계획 목록도 같은 철학을 따른다. 활성·종결·보류를 가리지 않고 모두 펼쳐 보이며, 접기/펼치기 토글을 두지 않는다. "한 화면에 다 들어가야 한다"는 가정 자체를 버리고 세로 스크롤로 훑는다.

상태 표시도 한 가지 시각 언어로 통일한다. 계획 전체 상태와 항목 상태가 같은 형태의 배지를 쓰고, 위치(헤더냐 항목 줄이냐)로만 층위를 구분한다. 디자이너는 배지 시스템 하나만 설계하면 된다. 흔하게 반복되는 "예정"은 노이즈가 되지 않게 가볍게 두고, 예외 상태(보류 등)가 눈에 들어오게 한다. 보류 항목은 은은하게 흐려지고 총액에서 빠진다.

### 문제 6 — 알리되 강요하지 않는다

항목이나 비용이 바뀌면 기존 동의는 무효가 되므로 "다시 서명받아야 한다"를 알려야 한다. 그러나 빨간 경고로 막지는 않는다. 동의를 다시 안 받았다고 진료를 멈추게 하지 않는다. 헤더의 서명 상태는 미서명 / 서명 완료(날짜와 함께) / 재서명 필요(동의 갱신 안내) 세 가지로, 가장 강한 "재서명 필요"도 경고가 아니라 안내 수준의 톤을 유지한다.

### 문제 7 — 자동으로 엮지 않는다

옵션을 바꾸면 단가가, 단가가 바뀌면 금액이 자동으로 따라 바뀌게 하는 것이 "똑똑해" 보인다. 그러나 직원이 직접 입력한 금액이 자동 계산에 덮어쓰이면 현장에서 신뢰를 잃는다. 그래서 자동 연동은 의도적으로 넣지 않았다. 시스템은 그릇만 제공하고, 의미와 값은 사용자가 채운다. 단, 총액 안에서 **할인액과 계약금액은 서로 연동**된다(한쪽을 고치면 다른 쪽이 합계+부가세를 기준으로 자동 계산). 이건 같은 한 가지 사실의 두 표현이라 연동이 자연스럽다.

## 4장. 프로토타입 화면 설명

위 맥락을 이해한 상태에서 본다. 화면은 좌측(차트 내용)과 우측(치료계획 탭)으로 나뉘고, 이 프로토타입은 우측을 다룬다. 우측은 위에서부터 세 덩어리다.

### (1) 헤더 — "누구의 계획인가" + 도구

좌측에는 계획에 귀속되는 정보를 둔다 — 담당의사, 상담직원, 상담일. 우측에는 상태·도구를 둔다 — 서명 상태 배지, 음성 녹음, 환자 표시 토글. 좌측은 "사람이 채우는 귀속 정보", 우측은 "시스템이 알려주거나 사용자가 조작하는 것"으로 성격이 갈린다.

이 정보들이 맨 위에 있는 이유는 시선 흐름과 맞기 때문이다. 사용자는 "이건 누구 계획인가 → 어떤 항목들인가 → 얼마인가" 순으로 읽는다.

### (2) 세부 목록 — 시술 항목들

치료계획을 구성하는 시술 항목이 한 줄씩 쌓인다. 한 줄은 치식 · 시술명/세부옵션 · 금액 · 메모 · 상태로 이뤄진다. 항목이 많아지면 이 목록 안에서 세로로 스크롤한다(헤더와 메타데이터는 고정).

- **치식**: 어느 치아인지를 치과 표준 표기(FDI)로 보여준다. 윗니/아랫니 2행, 좌우 치아 자리를 고정 좌표로 두고, 해당하는 치아 자리에만 번호를 찍는다. 한 항목이 여러 치아를 포함할 수 있고, 흩어진 위치도 가능하다.
- **시술명/세부옵션**: 시술명은 고정, 세부옵션은 눌러서 교체한다.
- **금액**: 그 항목의 최종 금액(원 단위 표시, 만원 단위 입력).
- **메모**: 그 항목에 대한 자유 메모(인라인 편집).
- **상태**: 예정/보류 등. 보류는 흐려지고 총액에서 빠진다.

목록 맨 아래에는 새 항목을 더하는 자리가 항상 있다.

### (3) 메타데이터 + 액션 — 계획 전체

좌측에는 계획 전체에 대한 메모, 우측에는 비용 명세(치료비 합계, 부가가치세, 할인액, 계약금액)를 둔다. 합계와 부가세는 자동으로 채워지는 읽기 전용 값이고(편집 칸과 배경색으로 구분), 할인액과 계약금액만 직접 입력하며 서로 연동된다.

오른쪽 끝에는 계획 단위 액션 버튼이 세로로 놓인다 — 치료계획 확정 · 환자 서명 · 인쇄 · 삭제. 채움 강도로 위계를 준다(확정이 가장 강하고, 삭제는 위험을 뜻하는 색을 절제해 쓴다). 삭제처럼 되돌릴 수 없는 동작은 확인 단계를 한 번 거친다.

| 시술명    | 변경 불가. 바꾸려면 삭제 후 추가                        | 옵션·단가 체계 정합성                |
| 세부옵션  | 시술명에 맞는 목록에서 선택                             | 자주 바뀌는 항목                     |
| 항목 할인 | 별도 필드 없음. 단가를 직접 수정                        | 환불 분쟁 시 항목 단가가 곧 가격     |
| 총액 할인 | 할인액↔계약금액 연동(합계+부가세 기준)                  | 같은 사실의 두 표현                  |
| 금액 단위 | 입력은 만원, 표시는 원 전체                             | 입력 부담과 표시 정확성 분리         |
| 자동 연동 | 옵션·단가→금액 자동 변경 안 함                          | 직접 입력값 보호                     |
| 상태      | 예정/진행중/완료/보류. 같은 시각 언어, 위치로 층위 구분 | 일관성, 배지 시스템 단일화           |
| 보류      | 흐림 처리 + 총액에서 차감                               | 삭제와 달리 이력·재개 여지           |
| 환자 표시 | 토글로 할인액 노출 제어. 할인 발생 시 자동 노출         | 환자 동석 화면의 할인 딜레마         |
| 동의·서명 | 변경 시 "재서명 필요" 안내. 강제하지 않음               | 알리되 강요 안 함 / 진료는 막지 않음 |
| 메모      | 인라인 편집. 전환 시 레이아웃 불변                      | 빠른 기록, 고밀도 화면 피로 최소화   |
| 목록      | 시간순 풀 노출, 접기 없음                               | 종이차트 철학                        |

### 용어

- **원장 / 봉직의(페이닥터)**: 개원의 / 고용된 의사
- **실장 / 데스크**: 접수·수납·상담을 맡는 직원
- **치식**: 치아 번호 표기(FDI 표준)
- **보류**: 환자가 당장 안 하기로 한 항목. 취소(삭제)와 구분
- **재서명 필요**: 항목·비용이 바뀌어 기존 동의가 무효가 된 상태

## 6장. 기타사항 — 미구현 / 열린 질문

이번 프로토타입에 담지 않았거나, 방향만 합의하고 넘긴 것들이다.

- **치식 변경 UI**: 항목의 치아를 다시 고르는 화면(모달 등)은 아직 없다. 표시만 한다.
- **항목 추가 시 치식 선택 강제**: 새 항목을 만들 때 치식부터 고르게 하는 흐름을 의도하나, 이번엔 빈 줄 생성까지만 했다.
- **서명 상태 연동**: 서명 배지의 상태는 아직 고정값이다. 실제로는 항목·비용 변경(재서명 트리거)과 하단 "환자 서명" 버튼, 그리고 동의 이력에 연결돼야 한다. 이 배지와 하단 버튼의 역할 관계를 먼저 정해야 한다.
- **단가 마스터 연동**: 옵션 선택 시 기본 단가가 자동으로 채워지고, 단가가 마스터와 다를 때 약하게 표시하는 규칙(원안에 있던)은 아직 붙이지 않았다. 자동 연동을 넣을지/어디까지 넣을지는 직접 입력값 보호와의 균형에서 다시 정해야 한다.
- **금액 검증**: 할인액·계약금액에 합계를 넘는 값을 넣으면 음수가 표시될 수 있다. 범위 제한이나 안내를 둘지 미정.
- **환자 전송**: 장기적으로 환자 핸드폰 전송이 1순위 예정이나 현 단계에서는 제외.
- **일정·첨부·수납**: 별도 영역이다. 치료계획에서는 결과만 반영되거나 참조만 한다.
